\begin{abstract}
TOPIC Connector is an Apache-2.0-licensed software toolkit for deploying and inspecting early-stage Eclipse Dataspace Components (EDC)-based connector stacks. It packages an EDC runtime with PostgreSQL persistence, an operator web interface, local file publication support, download-capture services, Nginx gateway routing, Docker Compose profiles, examples, and validation scripts. The software is motivated by municipal and geospatial data-sharing pilots, where teams need a reproducible connector baseline before investing in production hardening or cross-organizational certification. The current artifact validates stack health, local asset upload, and download-capture registration through executable smoke tests and continuous integration. ArcGIS authentication and item-upload paths are implemented as integration hooks, but live institutional ArcGIS validation and full cross-connector interoperability benchmarking remain outside the evidence reported here.
\end{abstract}

\section{Motivation and significance}
Data spaces require more than a connector runtime. A participant must configure routes, persistence, management access, publication workflows, policies, contract definitions, and transfer handling before data exchange can be operated by non-framework specialists. This gap is relevant in municipal settings, where data may appear as local files, geospatial services, or GIS-managed resources, and where the same technical baseline may need to be replicated across several organizations.

TOPIC Connector addresses this practical gap by packaging a reproducible EDC-based connector stack with the auxiliary services needed for early publication and transfer experiments. The artifact is not presented as a certified production connector or as a complete interoperability benchmark. Its purpose is narrower: to provide a public, inspectable, and executable baseline for researchers and engineers studying connector deployment, local asset publication, operator-facing management, and municipal-oriented GIS integration hooks.

The software can be reused outside the original EITEL project context by teams that need to evaluate EDC deployment patterns, compare connector packaging strategies, prototype data-space publication workflows, or teach connector operation without assembling each component manually. Compared with starting from a minimal EDC runtime, TOPIC Connector saves integration effort by providing a ready local stack, UI wiring, gateway configuration, local upload service, download sink, examples, and validation scripts.

\section{Related work}
European data-space policy frames trusted data sharing through instruments such as the European strategy for data, the Data Governance Act, and the Data Act~\cite{b1,b2,b3,b4}. IDS-RAM and Gaia-X provide architectural foundations for sovereignty, trust, usage control, and federated participation~\cite{b5,b6}. These references define the setting in which connectors operate, but they do not by themselves provide a complete institutional deployment package.

EDC is a major open framework for building dataspace services and connectors~\cite{b7}. Existing connector implementations differ in architecture, identity handling, and policy-language choices, as shown by surveys and implementation reports covering DSC, EDC, TRUE Connector, and Trusted Connector~\cite{b12,b13}. Urban and geospatial data-space literature further shows that governance, standardized ICT architectures, and spatial-data tooling affect adoption in public-sector environments~\cite{b14,b15,b16}. Recent work also explores service-oriented interaction models on top of EDC~\cite{b17}. TOPIC Connector is positioned below these broader research questions: it supplies a packaged software baseline that can be inspected, reproduced, and extended for early-stage municipal-oriented connector experiments.

The publication workflow supports remote URLs and locally uploaded files. A resource becomes data-space-ready only after it is represented as an EDC asset and linked to policy and contract-definition information. The UI exposes this workflow so that publication steps do not depend only on manual management-API calls.

The software also implements ArcGIS-oriented authentication and item-upload hooks. These hooks are included because municipal geospatial data is often curated or consumed in GIS platforms. In the current evidence, they should be interpreted as implemented integration paths, not as live production validation against an institutional ArcGIS Portal.

\section{Illustrative workflow}
A typical local experiment starts with a provider running the Docker Compose stack and opening the UI through the Nginx gateway. The provider uploads a GeoJSON file to the local-assets service, receives a controlled retrieval URL, registers the asset in the connector, and attaches policy and contract-definition information. A consumer-side workflow can then inspect catalog information, negotiate access, execute transfer, and capture the resulting payload in the download-sink service. The current release validates the local upload and download-capture parts of this workflow; full cross-connector negotiation is retained as experimental material.

\section{Reproducibility and validation}
The intended reproduction sequence is:

\begin{verbatim}
git clone https://github.com/krgroup/TOPIC-Connector.git
cd TOPIC-Connector
git checkout v1.0.11
cp .env.example .env
docker compose --env-file .env -f docker-compose.yaml up -d --build
./scripts/validate_stack.sh
./scripts/validate_local_asset_upload.sh
./scripts/validate_download_capture.sh
\end{verbatim}

The repository CI validates Docker Compose configuration, checks selected frontend JavaScript files, starts the local reproduction stack, and runs the three smoke checks. Table~\ref{tab:evidence} separates implemented capabilities from validated evidence.

The smoke tests intentionally focus on reproducibility and operational substrate: whether the stack starts, whether local files can become connector publication inputs, and whether transferred payloads can be captured with traceable identifiers. They do not establish full data-space interoperability, policy enforcement, performance, or production security certification.

\section{Impact}
TOPIC Connector is useful as a reproducible research and engineering baseline. It enables experiments on connector packaging, local asset onboarding, gateway routing, UI-mediated publication, and transfer observability without requiring each user to assemble an EDC stack and auxiliary services from scratch. The municipal and GIS framing is the motivating use case, but the same packaging approach can be reused in other public-sector or research pilots where controlled files, APIs, and operator-facing workflows must be connected to a data-space runtime.

The artifact also supports comparative work. Researchers can replace or extend components, compare the packaged workflow with a minimal EDC runtime, or use the repository as a teaching and demonstration environment for connector operation. It can support studies of EDC packaging strategies, reproducibility of connector deployments, and the practical separation between implemented connector capabilities and independently validated interoperability evidence. Its most immediate impact is therefore not production certification, but lowering the barrier to reproducible connector experimentation.

\section{Security scope and limitations}
The local reproduction profile uses placeholder tokens and is intended for local review. Historical and demo profiles may expose management or local-assets tokens to browser-visible variables. This is acceptable only in local demonstrations. Closed pilots should use authenticated gateway access and rotated secrets. Production deployments should hide privileged management operations behind an authenticated backend or gateway layer and should not expose privileged keys to browser clients.

The main limitation is validation depth. The current paper reports smoke tests rather than a longitudinal multi-organization study. It does not report live ArcGIS Portal validation, independent municipal operator evaluation, performance measurements, or cross-connector interoperability benchmarks. These are necessary next steps before making stronger claims about production municipal deployment.

\section{Conclusions}
TOPIC Connector provides a public, Apache-2.0-licensed toolkit for reproducing and inspecting an EDC-based connector stack with UI, local upload, download capture, gateway routing, examples, and validation scripts. The artifact is best understood as an early-stage municipal-oriented experimentation baseline. Its current evidence supports reproducible local deployment, local asset upload, and download-capture registration; broader claims about production operation, ArcGIS deployment, and cross-connector interoperability remain future work.
